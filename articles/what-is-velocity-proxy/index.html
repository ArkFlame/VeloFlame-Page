What is a Velocity proxy?

Velocity is a modern Minecraft proxy used to connect multiple backend servers behind one network entry point.

Velocity is a modern Minecraft proxy used to connect multiple backend servers behind one network entry point.

Direct answer

For most new networks, start with a modern proxy architecture, then add only the security and compatibility layers you can observe, tune, and maintain.

VeloFlame is positioned for owners who want a Velocity-style proxy with practical connection-defense controls built into the proxy layer: firewall behavior, rate limiting, VPN/proxy list checks, reconnect checks, logging, and allowlist workflows. That does not replace backend plugins. It reduces the amount of suspicious traffic that backend servers and plugins must handle.

What the proxy should own

A Minecraft proxy sits before backend servers. That location makes it the right place for routing, player forwarding, initial connection policy, and attack visibility. Backend Paper, Spigot, or Folia servers should still own gameplay, worlds, permissions, regions, economy, inventories, and plugin-specific behavior.

  • Reject obvious connection floods before backend login plugins allocate more work.
  • Keep backend servers reachable only through the trusted proxy path.
  • Log blocked connection reasons in a way staff can tune during real attacks.
  • Separate proxy plugins from Paper plugins so each layer solves the right problem.

Where VeloFlame fits

VeloFlame is not a generic backend plugin. It belongs at the proxy edge of the network. That matters because many server owners try to solve bot attacks after the traffic has already reached Paper. By then, the backend may already be spending time on handshake, login, plugin listeners, database checks, or authentication workflows.

If your current network is small, the main win is simpler operational control: fewer moving parts and clearer logs. If your network is public or growing, the win is earlier filtering and fewer backend symptoms during join bursts or bot pressure.

Operator checklist

  • Confirm all backend servers only accept traffic from the proxy or trusted private network.
  • Review forwarding mode, online-mode assumptions, and backend authentication plugins before going public.
  • Start with conservative rate limits, then tune with logs instead of guessing.
  • Keep a rollback path when migrating from BungeeCord, Waterfall, or plain Velocity.
  • Test from a clean client, a staff account, and a reconnect-heavy scenario before announcing the change.
